# PR11.x Series — Proposed Milestone Roadmap

This document is a **proposed** milestone series for discussion and refinement.
It is not wired into `execution/tracker.json`, `execution/dashboard.md`, CI, or
any gate script. No milestone described here is in progress or done.

Source material: `docs/post_pr10_scope.md` (PS-xxx ledger),
`docs/syntax_proposals.md` (15 syntax proposals),
`docs/frontend_architecture_baseline.md` (current accepted language surface),
and `docs/emitted_output_verification_matrix.md` (current proof coverage).

## Prerequisites

PR10.3, PR10.4, and PR10.5 should complete before any work described here
begins. Those milestones (already tracked in `execution/tracker.json`) close the
sequential proof-corpus expansion, parser/evidence hardening, and emitter
maintenance series respectively. All three depend on PR10.1.

Dependency chain: PR10.1 → PR10.3 / PR10.4 / PR10.5 → PR10.6 → PR10.7 → PR11.1+

---

## PR10.6: Minimal Developer Ergonomics

Limited tooling so syntax proposals can be evaluated with real feedback instead
of command-line-only compilation. Does not attempt full IDE support — that is
deferred until post-v1.0.

### Deliverables

**`safe build <file.safe>` — one-command compile wrapper.**
Python script calling `safec emit --ada-out-dir` followed by `gprbuild`. Produces
a compiled Ada executable from a single Safe source file. Follows the no-Python-
in-safec-runtime doctrine: this is a separate orchestration tool, not part of
`safec` itself.

**VSCode language grammar — static `.tmLanguage.json`.**
Keyword highlighting, comment and string coloring, bracket matching. Updated
when syntax changes (estimated ~20 minutes per update). No semantic analysis,
no type-aware highlighting.

**Diagnostics LSP shim — thin JSON-RPC process.**
Calls `safec check --diag-json` on save, maps `diagnostics-v0` output to LSP
`publishDiagnostics`. No completion, no hover, no rename, no go-to-definition.
Explicitly disposable — will be replaced by a real LSP server post-v1.0.

### Scope Exclusions

- No formatter
- No file watcher
- No code completion
- No package manager integration

## PR10.7: Rosetta Code Sample Corpus

A living code corpus under `samples/rosetta/` so every syntax change is
immediately visible across real programs rather than hand-crafted examples.

### Initial Corpus (12 programs)

**Arithmetic:**
- `fibonacci.safe`
- `gcd.safe`
- `factorial.safe`
- `collatz_bounded.safe`

**Sorting:**
- `bubble_sort.safe`
- `binary_search.safe`

**Data structures:**
- `linked_list_reverse.safe`
- `bounded_stack.safe`

**Concurrency:**
- `producer_consumer.safe`
- `prime_sieve_pipeline.safe`

**Numeric:**
- `trapezoidal_rule.safe`
- `newton_sqrt_bounded.safe`

### Gate and CI

`scripts/run_rosetta_corpus.py` validates the entire corpus: `safec check` →
`safec emit --ada-out-dir` → `gprbuild` compile. A CI job runs this on every PR.

### Growth Policy

The corpus grows as the language surface expands. Strings, case statements,
generics, and each new syntax feature unlock additional Rosetta tasks. New
programs are added to the corpus as each milestone lands.

### Syntax-Proposal UAT

The corpus enables mechanical syntax-proposal evaluation. When a syntax change
is proposed, the transformation is applied across all 12+ corpus programs,
compiled, and reviewed. This replaces ad-hoc example evaluation with systematic
acceptance testing across diverse program structures.

## PR11.2: Syntax Transformation

Implement the accumulated syntax proposals from `docs/syntax_proposals.md`.

### Proposals Included

| Syntax Proposal | Summary |
|-----------------|---------|
| Whitespace-Significant Blocks | Indentation-defined block structure as default mode |
| `pragma Strict` | Full Ada-style keyword-delimited syntax with closing labels |
| `returns` Keyword | Replace `return` in signatures with `returns` |
| `else if` Keyword | Replace `elsif` with `else if` |
| Optional Semicolons | Go-style lexer-inserted semicolons |
| `var` Declarations | Statement-local variable declarations (from Statement Labels, Loop Labels, and `var` Declarations proposal) |
| Capitalisation as Reference Signal | Uppercase initial = reference type; plus `move` keyword |
| Implicit Dereference | Drop `.all` explicit dereference (subsection of Capitalisation as Reference Signal) |

### Migration

- Existing test corpus mechanically transformed to new syntax.
- Rosetta corpus mechanically transformed and reviewed.
- Both old-syntax and new-syntax parsing may coexist during the transition if
  staging is chosen (see Open Questions).

## PR11.3: Numeric Model

Type system foundations for the integer and floating-point model.

### Scope

- Three-tier integer model with wide-intermediate overflow checking — this is a
  coupled, atomic change (the tiers and the overflow model must ship together).
- Simplified predefined type names: `integer`, `short`, `byte` (from the
  Simplified Predefined Type Names proposal).
- PS-028 (TBD-10): Resolution or explicit deferral with versioned policy for
  required ranges of predefined integer types.
- PS-030 (TBD-12): Resolution or explicit deferral with versioned policy for
  modular arithmetic wrapping semantics.

### Migration

- Rosetta corpus updated for new type names.
- Existing test corpus updated for new integer model.

---

## PR11.4: Artifact Contract Stabilization

Machine-interface stability for ecosystem consumers.

### Scope

- Compatibility policy for `diagnostics-v0`, `safei-v1`, `mir-v2`.
- Documented bump rules: what changes require a version increment, what changes
  are additive-only, what changes are breaking.
- Resolves PS-021 (TBD-08): stabilise and document interchange-format policy.

### Significance

This is the "tooling-ready contract freeze." After this milestone, external
tools can build against stable interfaces with documented compatibility
guarantees. The diagnostics LSP shim from PR10.6, for instance, can move from
"explicitly disposable" to "stable interface consumer."

---

## PR11.5: Monomorphic Library Layer

"Viable programs" before generics land.

### Scope

- Bounded string buffer (monomorphic, for built-in character/string types).
- Bounded containers for built-in types: bounded vector, bounded map, bounded
  set — no generics required, implemented as concrete types for specific element
  types.
- Copy-semantics-only restriction: no move-semantic container elements in this
  milestone.

### Rationale

Resolves the immediate "can't write real programs without collections" gap.
Programs can use concrete bounded containers now; these become generic
instantiations in PR11.6.

---

## PR11.6: Restricted Generics

The big capability milestone.

### Scope

- Generic package declarations and monomorphic instantiation.
- Standard generic containers: `Bounded_Vectors`, `Bounded_Maps`,
  `Bounded_Sets`.
- `String_Buffer` becomes a generic instantiation rather than a monomorphic
  special case.
- Copy-semantics-only element restriction for v1.0 (move-semantic elements
  deferred).
- Spec TBD register items related to generics resolved or explicitly
  version-targeted.

### Emitter-Based Instantiation

Follows the Emitter-Based Container Instantiation proposal from
`docs/syntax_proposals.md`: generic instantiation produces monomorphic Ada
code at emit time rather than requiring Ada-side generic machinery.

## Ordering Rationale

**Ergonomics before syntax** — PR10.6 delivers `safe build`, syntax
highlighting, and diagnostics-on-save so that syntax proposals can be evaluated
with real tooling feedback instead of command-line-only compilation.

**Corpus before syntax** — PR10.7 creates a diverse program corpus so that
syntax proposals are evaluated across real programs, not just hand-crafted
examples.

**Parser completeness before syntax transformation** — PR11.1 adds strings and
case statements, which unlock real programs. Those programs then get transformed
in PR11.2, providing better syntax-change evaluation.

**Numeric model after syntax** — PR11.3 follows PR11.2 because syntax changes
are surface-only (they don't affect the proof model), while type-system changes
affect the proof model and need stable syntax to build on.

**Artifact contracts before library** — PR11.4 stabilises machine interfaces
before PR11.5 introduces library types that external tools will consume. Library
consumers need stable interfaces.

**Monomorphic library before generics** — PR11.5 ships useful bounded containers
sooner, and the concrete implementations serve as test cases for the generic
design in PR11.6.

**Generics before spec freeze** — PR11.6 introduces generics, which affect the
type system. The spec cannot be baselined until the type system is complete.

## Open Questions

Decisions that need resolution before committing to this plan:

1. **Should PR11.2 implement all syntax proposals at once or stage them?**
   Implementing everything in one milestone is simpler to test (one migration)
   but higher risk. Staging (e.g., whitespace blocks first, then semicolons,
   then capitalisation) allows incremental corpus evaluation but requires
   multiple migration passes.

2. **Should the Capitalisation as Reference Signal proposal be part of PR11.2 or
   deferred?** This proposal has the deepest semantic implications (it changes
   how reference types are recognized). It could be staged separately from the
   purely syntactic changes.

3. **Should `pragma Strict` be implemented in the same milestone as the default
   syntax, or separately?** Implementing both together ensures they stay
   consistent, but adds scope. Implementing the default mode first and adding
   `pragma Strict` later risks divergence.

4. **What is the v1.0 target date (if any)?** The milestone sequence is defined
   but no timeline is attached. A target date would help prioritize "not
   scheduled" items and resolve the TBD register.
