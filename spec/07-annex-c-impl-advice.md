# Annex C — Implementation Advice

1. This annex provides non-normative implementation advice for conforming Safe implementations. Requirements in this annex use "should" rather than "shall" — they are recommendations, not requirements.

---

## C.1 Symbol File Format

2. The symbol file format is implementation-defined. The following advice is provided for implementers.

3. The symbol file should be a compact binary format, not a text format. Binary symbol files are faster to read and write, and their compactness speeds up incremental recompilation of large projects.

4. The symbol file should contain a version identifier and a content hash. The version identifier allows the compiler to reject symbol files from incompatible compiler versions. The content hash enables change detection for incremental recompilation — if the hash has not changed, dependents need not be recompiled.

5. The symbol file should include the following information for each public declaration:

| Declaration Kind | Exported Information |
|---|---|
| Type (non-opaque) | Full type definition, size, alignment |
| Type (opaque) | Type name, size, alignment (no field layout) |
| Subtype | Constraint, parent type reference |
| Subprogram | Full signature, `Global` set, `Depends` set |
| Constant | Type, value (if static) |
| Object | Type, initial value constraint |
| Channel | Element type, capacity |

6. For opaque types (`public type T is private record`), the symbol file should export the size and alignment of the type but not the field layout. This enables clients to declare variables of the type (the compiler knows how much stack or heap space to allocate) without accessing the internal structure.

7. The symbol file should be stored alongside the source file with a `.safi` extension (Safe Interface). For example, `sensors.safe` produces `sensors.safi`.

## C.2 C Emission Quality

8. The emitted C99 code should be human-readable. Variable names, function names, and type names should be derived from the Safe source identifiers, not from internal compiler representations.

9. **Naming conventions:** The following naming scheme is recommended:

| Safe Entity | C Name |
|---|---|
| Package `P` | Prefix `p__` on all entities |
| Type `T` in package `P` | `p__t_t` (type definition) |
| Function `F` in package `P` | `p__f` (function definition) |
| Variable `V` in package `P` | `p__v` (global variable) |
| Channel `C` in package `P` | `p__c` (channel struct) |
| Task `T` in package `P` | `p__t_task` (task entry point) |
| Local variable `X` | `x` (within the enclosing function) |
| Parameter `P` | `p` (function parameter) |

10. For hierarchical package names (`Parent.Child`), the C prefix should use double underscores as separators: `parent__child__entity`.

11. **Indentation:** The emitted C should use consistent indentation (4 spaces recommended) and include blank lines between function definitions for readability.

12. **Comments:** The emitter should generate comments identifying the Safe source location for each emitted construct:

```c
/* sensors.safe:42 -- function Get_Reading */
int32_t sensors__get_reading(int32_t channel) {
    ...
}
```

13. **Header files:** The emitter should generate a `.h` header file for each package, containing:
- Type definitions for public types
- Function prototypes for public subprograms
- External variable declarations for public objects
- Opaque type declarations as `typedef struct name_t name_t;` (incomplete type) for opaque types

14. This enables C code to interface with compiled Safe packages through standard C header inclusion.

---

## C.3 Ada/SPARK Emission Quality

15. The emitted Ada/SPARK code should be human-readable and closely mirror the Safe source structure.

16. **Naming:** Safe identifiers should be preserved exactly in the emitted Ada. No name mangling should be applied.

17. **Formatting:** The emitted Ada should follow standard Ada formatting conventions: 3-space indentation, blank lines between declarations, standard casing (types in Title_Case, reserved words in lowercase).

18. **Comments:** The emitter should include a header comment in each emitted file identifying it as generated code:

```ada
--  Generated by safe --emit-ada from sensors.safe
--  Do not edit — regenerate from Safe source.
pragma SPARK_Mode;
```

19. **Annotation placement:** `Global` and `Depends` aspects should appear on the subprogram declaration in the `.ads` file, using the `with` aspect syntax. `Initializes` should appear on the package declaration.

20. The emitted Ada should compile without warnings under `gnatmake -gnatwae` (all warnings, warnings as errors).

---

## C.4 Incremental Recompilation

21. A conforming implementation should track compilation dependencies and support incremental builds.

22. **Dependency tracking:** The compiler should record, for each compiled package, the set of symbol files it read during compilation. When any of those symbol files changes, the package must be recompiled.

23. **Change detection:** The compiler should use content hashing (e.g., SHA-256) on symbol files rather than timestamps. This avoids unnecessary recompilation when a dependency's source file is touched but its public interface has not changed.

24. **Build system integration:** The compiler should support a build-file format (e.g., a project file similar to GNAT's `.gpr` files) that specifies:
- Source directories
- Output directory
- Target platform
- Compiler options

25. Alternatively, the compiler should support a file-system convention where all `.safe` files in a directory constitute a project, and dependencies are resolved by finding `.safi` symbol files in a search path.

---

## C.5 Diagnostic Messages

26. Diagnostic messages should follow the format:

```
file:line:column: severity: message [Safe §reference]
```

27. **Severity levels:**
- `error` — the program is rejected
- `warning` — the program is accepted but may have issues
- `note` — additional information about a previous error or warning

28. **Rule references:** Every error diagnostic for a Safe-specific rule should include a bracketed reference to the relevant section of this specification. Examples:

```
sensors.safe:42:15: error: right operand of "/" has type Integer whose range includes zero [Safe §2.8.3]
sensors.safe:50:20: error: dereference of access type Node_Ptr which may be null; use Node_Ref [Safe §2.8.4]
sensors.safe:60:5: error: generics are not permitted [Safe §2.1.10]
sensors.safe:70:5: error: variable Cal_Offset accessed by both task Sampler and task Processor [Safe §4.5]
```

29. **Suggestions:** For Silver-by-construction rule violations, the diagnostic should suggest the correct pattern:

```
sensors.safe:42:15: error: right operand of "/" has type Integer whose range includes zero [Safe §2.8.3]
sensors.safe:42:15: note: consider using subtype Positive (range 1 .. Integer.Last) or declaring a custom nonzero subtype
```

30. **Ownership diagnostics:** For ownership violations, the diagnostic should identify the ownership state and the conflicting operation:

```
lists.safe:30:5: error: access value P was moved at line 28; cannot be read before reassignment [Safe §2.3.1]
lists.safe:35:5: error: access value Q is frozen (borrowed by parameter at line 33); cannot be accessed [Safe §2.3.2]
```

## C.6 Defense-in-Depth Checks

31. Although the Silver-by-construction rules guarantee that certain runtime checks are provably safe, implementations should emit runtime checks in the `--emit-c` backend as defense-in-depth. The D27 rules guarantee that these checks never fire in a correct program, but they provide an additional safety net against compiler bugs.

32. Defense-in-depth checks should be controllable via a compiler flag (e.g., `--checks=full` vs. `--checks=minimal`). The default should be `--checks=full`.

33. The runtime check handlers should be marked `__attribute__((cold))` or equivalent to minimize the performance impact of the check code on the hot path.

---

## C.7 Compilation Speed

34. The single-pass compilation model (D3) should result in fast compilation. As a guideline, a conforming implementation should compile at least 50,000 lines of Safe source per second on a modern processor (amd64, 3+ GHz).

35. Symbol file reading should be optimized — loading a binary symbol file should be significantly faster than parsing the source file it represents.

---

## C.8 Error Recovery

36. The parser should implement error recovery to report multiple errors per compilation, rather than stopping at the first error. Recommended strategies:
- **Synchronization tokens:** On error, skip to the next semicolon, `end`, or `begin` keyword and resume parsing.
- **Maximum error count:** Stop after 50 errors to avoid cascading error floods.

37. The parser should not attempt to correct errors automatically — it should report the error, recover, and continue parsing to find additional errors.

---

## C.9 Cross-Compilation

38. The compiler should support cross-compilation from any host platform to the target platforms (OpenBSD/amd64, OpenBSD/arm64). This is inherent in the C emission model — the compiler emits C99 source, which is compiled by the target platform's C compiler.

39. For `--emit-ada`, cross-compilation is handled by GNAT's cross-compilation support.

---

## C.10 Testing Recommendations

40. A conforming implementation should include a test suite that exercises:
- Every retained feature from 8652:2023
- Every excluded feature (testing that it is correctly rejected)
- Every Silver-by-construction rule (both positive and negative tests)
- Every channel and task construct
- Every ownership model pattern (move, borrow, observe, scope-exit deallocation)
- The `--emit-ada` backend (comparing GNATprove results against expected outcomes)
- The `--emit-c` backend (compiling and running the emitted C on target platforms)

41. The test suite should include at least one test for each numbered paragraph in Section 2 (Restrictions) to verify that every exclusion is enforced.
